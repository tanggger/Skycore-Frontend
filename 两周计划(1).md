# 两周执行计划：低空无人机蜂群通信推演与对抗增强

## 1. 目标

本阶段不是重做前端，也不是重做后端底座，而是在现有系统基础上完成老师提出的增量增强，使系统从“通信仿真平台”进一步提升为：

- 面向多场景环境的无人机蜂群通信推演平台
- 支持合作场景下的通信规划与稳健性恢复
- 支持非合作场景下的通信观测、链路推断、干扰破坏与效果评估

本阶段默认前提：

- 前端主工作台已基本完成
- 核心后端仿真主程序已在 `scratch/uav_resource_allocation.cc` 中实现
- 当前工作重点是新增能力的组织、接入、展示与收口

## 2. 总体主线

本阶段按 3 条主线推进，不再把功能零散拆开：

| 主线 | 核心问题 | 最终产出 |
|---|---|---|
| 场景环境主线 | 不同地图/地形/障碍如何影响通信质量和拓扑连续性 | 多场景环境建模与前端可视化 |
| 合作场景主线 | 我方已知网络结构时，如何进行通信规划、模式切换和快速恢复 | 稳健性、时延、恢复效果展示 |
| 非合作场景主线 | 我方仅能观测到对方少量且带噪声通信数据时，如何推断其链路与通信模式并实施干扰 | 观测-推断-破坏-评估完整闭环 |

## 3. 范围定义

### 3.1 本阶段必须完成

- 支持至少 3 类典型场景的环境影响建模
- 明确合作场景与非合作场景的流程差异
- 非合作场景下形成“观测 -> 去噪 -> 链路推断 -> 关键节点/链路识别 -> 干扰/破坏 -> 效果评估”闭环
- 合作场景下形成“环境扰动/节点失效 -> 通信策略响应 -> 网络恢复 -> 指标对比”闭环
- 所有新增能力都能在现有前端中被看见和解释

### 3.2 本阶段可以降级实现

- 场景类型不求全面，只需先覆盖最能体现差异的几类
- 链路推断先做可解释、可演示版本，不追求最复杂算法
- 破坏推荐先做规则型推荐，不必做完整智能决策
- 通信模式先只做中心化与分布式两类

### 3.3 本阶段不作为优先目标

- 大规模算法调研写作
- 专利文本撰写
- 全量竞品深度报告
- 全场景、全参数、全模式的一次性完成

## 4. 三条主线的实施内容

### 4.1 场景环境主线

目标：

- 用统一的场景文件和地图环境来影响通信质量
- 强化“真实地图自适应生成通信环境”的能力
- 让前端沙盘中的环境变化真正对应后端通信变化

建议优先支持的场景：

- 城市高楼场景
- 森林遮挡场景
- 湖泊/空旷场景
- 野地/低遮挡场景

环境侧重点：

- 路损变化
- NLOS 影响
- 干扰强弱
- 连通距离变化
- 拓扑连续性变化

前端对应展示位置：

- 场景编辑器
- 中间 3D 沙盘
- 终端日志
- 左侧全局指标

### 4.2 合作场景主线

目标：

- 在已知我方网络结构的前提下进行通信规划
- 支持中心化和分布式两类通信模式
- 支持环境扰动和节点失效后的快速恢复

重点展示：

- 通信模式切换
- 响应速度
- 时延变化
- 连通率恢复
- 任务持续能力

前端对应展示位置：

- 左侧参数配置区
- 左侧全局指标区
- 右侧时序图
- 底部回放条
- 单机详情面板

### 4.3 非合作场景主线

目标：

- 我方无法直接知道对方真实通信拓扑
- 只能基于少量、局部、带噪声的通信观测数据进行推断
- 在推断基础上识别关键节点/关键链路并实施干扰或破坏

核心闭环：

1. 观测对方通信数据
2. 去噪与数据清洗
3. 恢复对方通信链路
4. 给每条边分配概率/可信度
5. 识别关键节点与关键链路
6. 实施干扰、压制或打掉节点
7. 评估对方网络受损程度和恢复能力

前端对应展示位置：

- 中间沙盘链路颜色与状态
- 右侧拓扑/指标区
- 单机详情面板
- 终端日志
- 回放分析

## 5. 两周时间表

### 第 1 阶段：第 1-3 天

主题：统一场景环境层

目标：

- 把场景类型、环境影响、地图输入方式统一起来
- 明确合作/非合作两类模式共享的环境底座

输出：

- 场景分类清单
- 各场景通信影响参数定义
- 与前端场景编辑器和沙盘的映射关系

验收标准：

- 前端可切换场景并明显看到环境差异
- 后端输出中能体现环境对通信的影响

### 第 2 阶段：第 4-6 天

主题：合作场景闭环

目标：

- 基于已知网络结构完成通信模式与恢复流程
- 支持中心化与分布式模式对比

输出：

- 合作场景下的通信策略演示链
- 节点失效/环境扰动后的恢复指标
- 响应速度和时延对比

验收标准：

- 前端能完整展示合作场景下的通信规划与恢复过程
- 能对比不同通信模式下的效果

### 第 3 阶段：第 7-11 天

主题：非合作场景核心闭环

目标：

- 完成从观测数据到链路推断，再到破坏评估的主链路

输出：

- 对方通信观测数据处理能力
- 去噪后的链路推断结果
- 边概率或可信度展示
- 关键节点/关键链路识别
- 干扰注入或节点打击后的效果评估

验收标准：

- 前端能看到“推断出的对方链路”
- 能展示“打击前后网络变化”
- 能形成老师可理解的非合作对抗故事线

### 第 4 阶段：第 12-14 天

主题：联调、收口、展示

目标：

- 固定演示方案
- 清理输入输出口径
- 保证老师汇报时能够顺畅说明

输出：

- 一条合作场景 demo
- 一条非合作场景 demo
- 一份结果图和指标口径说明
- 一份汇报讲述顺序

验收标准：

- 全链路可演示
- 每个阶段都能在前端上看到结果
- 能用统一语言解释系统能力与创新点

## 6. 前端信息架构重构方案

上一版前端方案的问题在于：默认把合作场景和非合作场景都压在同一个工作台里，虽然技术上能实现，但逻辑上不清晰，老师和评委会很难在短时间内理解“现在到底在做哪类任务”。

因此，本阶段前端不应继续沿着“所有能力都叠加到一个单页主视图”推进，而应该重构为：

> 统一入口页 + 两种模式工作区 + 共享底层组件能力

也就是说：

- 合作场景和非合作场景必须是两种明确分开的任务模式
- 二者共享地图、沙盘、回放、图表、单机详情等底层能力
- 但不能在同一个工作视图里默认混放所有控制项和分析项

## 6A. 前端总体结构

前端最终应拆成 3 层结构：

### 6A.1 第一层：模式入口页

这是进入系统后的第一层，不直接进入推演页面，而是先选择任务模式。

入口页至少包含：

- 场景环境入口
  - 城市
  - 森林
  - 湖泊/空旷
  - 野地
- 任务模式入口
  - 合作场景
  - 非合作场景
- 最近任务 / 典型案例入口
- CSV 离线复盘入口
- OSM/地图编辑入口

入口页的作用：

- 先告诉用户“这次要解决的是什么问题”
- 再进入对应工作流
- 避免一上来就进入一个过于复杂的大杂烩页面

### 6A.2 第二层：合作场景工作区

合作场景工作区专门服务“我方已知网络结构”的任务，不要混入非合作推断逻辑。

它的核心问题是：

- 我方如何规划通信拓扑
- 选择中心化还是分布式通信模式
- 在环境扰动和节点失效后如何快速恢复

合作场景工作区应默认强调：

- 任务配置
- 通信模式
- 稳健性指标
- 恢复过程
- 我方网络状态

### 6A.3 第三层：非合作场景工作区

非合作场景工作区专门服务“我方无法直接知道对方真实通信链路”的任务。

它的核心问题是：

- 我方只能观测到少量且带噪声的数据
- 如何推断对方通信模式和连接方式
- 如何识别关键节点 / 关键链路
- 如何实施干扰、压制或节点打击

非合作场景工作区应默认强调：

- 观测条件
- 推断结果
- 可信度
- 对抗动作
- 破坏效果

## 6B. 合作场景工作区设计

### 6B.1 页面定位

合作场景页不是“指挥敌我对抗”，而是“我方蜂群任务推演与恢复控制台”。

页面主叙事应围绕：

- 场景环境如何影响我方通信
- 我方通信模式如何配置
- 当环境恶化或节点失效时如何恢复

### 6B.2 页面布局建议

合作场景工作区可以保留现有三栏布局，但语义要重新组织：

- 左侧：任务与模式配置
- 中间：我方蜂群 3D 沙盘
- 右侧：性能与恢复分析
- 底部：事件回放条

### 6B.3 左侧内容

左侧只保留合作场景真正需要的配置：

- 场景类型
- 蜂群规模
- 阵型
- 起点终点
- 场景编辑器入口
- 通信模式
  - 中心化
  - 分布式
- 恢复策略开关
- 环境强度参数

不应默认放入：

- 红蓝对抗设置
- 敌方观测噪声参数
- 推断链路控制项

### 6B.4 中间沙盘内容

合作场景沙盘强调：

- 我方无人机编队
- 我方真实链路
- 环境分区影响
- 节点失效和恢复过程
- 路由/通信模式切换效果

颜色语义建议：

- 正常链路：青色
- NLOS 风险：橙色
- 冲突：红色
- 恢复中：黄色或脉冲高亮
- 失效节点：灰色

### 6B.5 右侧分析内容

合作场景右侧重点不放“推断”，而放：

- PDR
- 吞吐量
- P99 时延
- 连通率
- 恢复时间
- 响应速度
- 中心化 vs 分布式对比

### 6B.6 单机详情内容

合作场景单机详情应突出：

- 角色和位置
- 当前工作信道
- 邻居数量
- 当前模式
- 受干扰程度
- 是否参与恢复

## 6C. 非合作场景工作区设计

### 6C.1 页面定位

非合作场景页不是一般通信监控页面，而是“观测、推断、干扰、评估”的对抗工作台。

页面主叙事应围绕：

- 只能看到部分对方通信数据
- 这些数据是噪声化的、不完整的
- 需要先推断再打击

### 6C.2 页面布局建议

非合作场景工作区不适合照搬合作页的三栏逻辑，建议改成更偏“侦察-判断-打击”的布局：

- 左侧：观测与推断配置
- 中间：红蓝对抗沙盘
- 右侧：推断结果与破坏评估
- 底部：事件时间轴

### 6C.3 左侧内容

左侧应只放非合作场景真正需要的控制项：

- 观测比例
- 噪声等级
- 观测窗口长度
- 是否启用链路推断
- 推断策略选择
- 干扰方式选择
  - 干扰注入
  - 节点打击
  - 链路压制
- 破坏目标选择

不应默认放入：

- 我方通信模式细节配置
- 合作恢复策略细节

### 6C.4 中间沙盘内容

非合作场景沙盘要体现“认知不完整”：

- 我方节点与对方节点明确区分
- 对方真实链路不直接全部展示
- 重点展示推断链路
- 不同概率的推断链路要有不同视觉层级
- 关键节点、关键边、打击目标要重点高亮

颜色语义建议：

- 我方链路：青色
- 对方已确认链路：白蓝色
- 对方推断链路：虚线或半透明色
- 高概率链路：亮黄
- 低概率链路：淡黄或灰黄
- 干扰中的目标：红色脉冲
- 已受损链路：暗红或断裂效果

### 6C.5 右侧分析内容

非合作场景右侧要突出 3 组内容：

- 观测数据质量
  - 噪声等级
  - 可观测比例
  - 有效样本数
- 推断结果
  - 推断链路数
  - 平均边概率
  - 关键节点识别结果
  - 推断稳定度
- 破坏效果
  - 打击前后连通率变化
  - 目标网络受损程度
  - 恢复时间
  - 脆弱性评估

### 6C.6 单机详情内容

非合作场景单机详情不应和合作页完全相同，应强调：

- 该节点属于我方还是对方
- 对该节点的观测程度
- 是否为推断关键节点
- 与其关联的高概率边
- 当前受干扰或被打击状态

## 6D. 共享组件与共享层

合作和非合作不应该做成两套完全独立系统，而应共享底层能力：

- 3D 沙盘渲染引擎
- 回放引擎
- CSV / API 数据加载
- 场景编辑器
- 单机详情弹窗
- 通用图表组件
- 事件时间轴能力

前端真正需要新增的是：

- 模式入口层
- 合作工作区容器
- 非合作工作区容器
- 两套不同的信息架构

## 6E. 前端增量开发顺序

### 第一步：先做入口层

先把“单一大页面”改成“入口 + 模式工作区”。

最小实现要求：

- 进入系统先选模式
- 可以从入口进入合作页或非合作页
- 可以从工作区返回入口

### 第二步：再做合作页收口

把现有页面里合作场景需要的内容收束到合作工作区：

- 保留任务配置
- 保留我方 QoS 和恢复指标
- 去掉非合作推断项

### 第三步：再做非合作页独立化

把非合作需要的内容单独组织，不再夹杂在合作页里：

- 观测
- 去噪
- 推断
- 干扰
- 破坏评估

### 第四步：最后做共享细节统一

统一：

- 状态命名
- 颜色语义
- 事件命名
- 模式文案
- 图例和指标口径

## 6F. 前端模块增量任务表

| 模块 | 调整方向 | 最终角色 |
|---|---|---|
| App / 顶层容器 | 从单一主页面改成模式入口 + 工作区切换 | 前端任务总入口 |
| 入口页 | 新增 | 模式选择与案例入口 |
| 合作工作区 | 在现有页面基础上裁剪并强化 | 我方通信规划与恢复控制台 |
| 非合作工作区 | 在现有页面基础上拆出并重组 | 观测、推断、干扰、评估工作台 |
| TopBar | 增加模式、场景、任务状态摘要 | 全局态势栏 |
| LeftPanel | 按模式拆分配置项 | 各模式专属控制区 |
| CenterSandbox | 支持两种语义模式 | 共享核心沙盘 |
| RightPanel | 按模式切换内容重点 | 合作分析 / 推断分析双形态面板 |
| PlaybackBar | 增强事件时间轴 | 共用事件回放能力 |
| UAVDetail | 根据模式切换显示重点 | 单机级解释面板 |
| ScenarioEditor | 保持共享 | 场景构建与地图入口 |

## 6G. 前端两周排期

| 时间 | 前端重点 | 目标 |
|---|---|---|
| 第 1-2 天 | 搭出模式入口层与页面切换结构 | 解决“逻辑混乱”的根问题 |
| 第 3-4 天 | 收口合作工作区 | 让合作场景成为独立完整工作流 |
| 第 5-7 天 | 拆出非合作工作区 | 让非合作场景形成独立工作流 |
| 第 8-10 天 | 强化两种模式下的沙盘语义和右侧分析内容 | 让两类任务视觉与逻辑明显不同 |
| 第 11-12 天 | 增强回放、事件时间轴、模式化单机详情 | 让老师能跟着过程理解系统 |
| 第 13-14 天 | 统一文案、图例、颜色与展示口径 | 收口成答辩版本 |

## 6H. 前端最终验收标准

最终前端必须满足以下逻辑标准：

- 用户进入系统后，先选任务模式，而不是直接掉进一个信息过载的主页面
- 合作场景和非合作场景拥有明确分开的工作流
- 两个工作区虽然共享底层组件，但在控制项、分析内容和叙事重点上明显不同
- 老师或评委在 10 秒内能判断“当前看到的是合作任务还是非合作任务”
- 前端不再是功能堆叠，而是结构清晰的指挥舱体系

一句话总结新的前端增量开发目标：

> 不再把所有能力塞进一个页面，而是构建“统一入口 + 合作工作区 + 非合作工作区”的双模式指挥舱前端。

## 7. 关键里程碑

| 里程碑 | 时间 | 判定标准 |
|---|---|---|
| 场景环境可视化完成 | 第 3 天 | 不同场景能在沙盘和指标上体现差异 |
| 合作场景恢复闭环完成 | 第 6 天 | 可展示已知网络下的规划与恢复 |
| 非合作场景推断闭环完成 | 第 11 天 | 可展示观测、推断、破坏、评估 |
| 完整 demo 收口完成 | 第 14 天 | 合作/非合作两条演示链都可运行 |

## 8. 执行原则

- 每天做的内容都必须能映射到现有前端中的某个展示位置
- 优先做“老师一眼能看懂”的能力，不优先做最复杂算法
- 先保证主链路跑通，再逐步增强细节
- 两周内只追求“完整闭环增强版”，不追求“所有想法一次做完”

## 9. 最终交付形态

本阶段结束时，系统应至少具备以下可演示能力：

- 多环境场景影响通信质量与拓扑连续性
- 合作场景下的通信规划与快速恢复
- 非合作场景下基于观测数据的链路推断
- 人工干扰/节点打击后的对抗展示
- 恢复能力、时延、响应速度等核心指标展示

一句话总结本阶段目标：

> 在现有系统上完成“场景环境 + 合作恢复 + 非合作推断对抗”三条主线的最小可演示闭环。
